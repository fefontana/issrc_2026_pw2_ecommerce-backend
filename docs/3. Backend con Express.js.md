**Partes Fundamentales de un Backend Web (General)**

1.  **Manejo de Peticiones HTTP (Entrada/Salida Web):**
    *   **General:** Recibir peticiones HTTP (GET, POST, PUT, DELETE, etc.) del cliente, interpretar la URL, los encabezados (headers), el cuerpo (body) de la petición.
    *   **General:** Enviar respuestas HTTP al cliente, incluyendo códigos de estado (200 OK, 404 Not Found, 500 Internal Server Error, etc.), encabezados y el cuerpo de la respuesta (HTML, JSON, XML, etc.).

2.  **Enrutamiento (Routing):**
    *   **General:** Determinar qué código o lógica debe ejecutarse en función de la URL y el método HTTP de la petición. Es como el "despachador" de tráfico.

3.  **Middleware:**
    *   **General:** Funciones que se ejecutan secuencialmente durante el ciclo de vida de una petición-respuesta. Pueden modificar la petición, la respuesta, ejecutar código, terminar el ciclo o pasar el control a la siguiente función en la pila. Usos comunes: logging, autenticación, parseo de cuerpos de petición, manejo de errores.

4.  **Lógica de Negocio (Controladores/Servicios):**
    *   **General:** El "corazón" de la aplicación. Contiene las reglas, cálculos y procesos que definen cómo la aplicación responde a las peticiones y manipula los datos para cumplir los objetivos del sistema.

5.  **Acceso y Manipulación de Datos (Modelos/Capa de Datos):**
    *   **General:** Interacción con la(s) fuente(s) de datos (bases de datos, archivos, APIs externas). Esto incluye crear, leer, actualizar y eliminar datos (CRUD).
    *   **General:** A menudo implica la definición de "modelos" que representan la estructura de los datos.

6.  **Autenticación y Autorización:**
    *   **General:**
        *   **Autenticación:** Verificar la identidad de un usuario (¿quién eres?).
        *   **Autorización:** Determinar qué acciones puede realizar un usuario autenticado (¿qué tienes permitido hacer?).

7.  **Manejo de Errores:**
    *   **General:** Capturar errores que ocurren durante el procesamiento de una petición y responder adecuadamente al cliente (ej., con un código de error y un mensaje útil) y/o registrarlos para depuración.

8.  **Validación de Datos:**
    *   **General:** Asegurar que los datos recibidos del cliente (o de otras fuentes) sean correctos, completos y tengan el formato esperado antes de procesarlos o almacenarlos.

9.  **Configuración:**
    *   **General:** Gestionar variables y parámetros que pueden cambiar entre diferentes entornos (desarrollo, pruebas, producción), como credenciales de bases de datos, claves de API, puertos, etc.

10. **Seguridad (General):**
    *   **General:** Proteger contra vulnerabilidades comunes (XSS, CSRF, inyección SQL, etc.), manejo seguro de sesiones, encriptación de datos sensibles.

11. **Logging y Monitoreo:**
    *   **General:** Registrar eventos importantes, errores, y métricas de rendimiento para depuración, auditoría y análisis del comportamiento del sistema.

**Implementación Específica en un Backend con Express.js (Node.js)**

Ahora veamos cómo estos conceptos se traducen cuando usamos Express.js:

1.  **Manejo de Peticiones HTTP y Servidor Web:**
    *   **Node.js:** Proporciona el módulo `http` nativo para crear servidores HTTP.
    *   **Express.js:**
        *   `const express = require('express');`
        *   `const app = express();` (Crea una instancia de la aplicación Express)
        *   `app.listen(port, () => { ... });` (Inicia el servidor y escucha peticiones en un puerto)
        *   Los objetos `req` (petición) y `res` (respuesta) que Express proporciona a las funciones de enrutamiento y middleware son extensiones de los objetos `http.IncomingMessage` y `http.ServerResponse` de Node.js, con métodos y propiedades adicionales muy útiles.

2.  **Enrutamiento (Routing):**
    *   **Express.js:** Es una de sus características principales y más potentes.
        *   `app.METHOD(PATH, HANDLER)`:
            *   `app.get('/users', (req, res) => { /* ... */ });`
            *   `app.post('/products', (req, res) => { /* ... */ });`
        *   Parámetros de ruta: `app.get('/users/:userId', (req, res) => { const userId = req.params.userId; });`
        *   Queries de ruta: `req.query` para acceder a parámetros como `?name=John`
        *   **Express Router (`express.Router()`):** Para modularizar las rutas en archivos separados, haciendo el código más organizado.
            ```javascript
            // routes/userRoutes.js
            const router = express.Router();
            router.get('/', getAllUsers);
            router.post('/', createUser);
            module.exports = router;

            // app.js
            const userRoutes = require('./routes/userRoutes');
            app.use('/api/users', userRoutes); // Monta el router en una ruta base
            ```

3.  **Middleware:**
    *   **Express.js:** Fundamental para su funcionamiento. Los middleware son funciones con la firma `(req, res, next)`.
        *   **Middleware a nivel de aplicación:** `app.use((req, res, next) => { /* ... */ next(); });` (Se ejecuta para cada petición)
        *   **Middleware a nivel de router:** `router.use(...)`
        *   **Middleware específico para una ruta:** `app.get('/ruta', miMiddleware, miHandler);`
        *   **Middleware incorporados:**
            *   `express.json()`: Para parsear cuerpos de petición en formato JSON (convierte `req.body` en un objeto JavaScript).
            *   `express.urlencoded({ extended: true })`: Para parsear cuerpos de petición URL-encoded (formularios HTML tradicionales).
            *   `express.static('public')`: Para servir archivos estáticos (CSS, JS del cliente, imágenes).
        *   **Middleware de terceros:** Numerosos paquetes NPM (ej: `morgan` para logging, `cors` para habilitar Cross-Origin Resource Sharing, `helmet` para seguridad básica, `body-parser` aunque ahora sus funciones principales están en `express.json()` y `express.urlencoded()`).
        *   **Middleware de manejo de errores:** Un tipo especial de middleware con cuatro argumentos `(err, req, res, next)` que se define al final.

4.  **Lógica de Negocio (Controladores/Servicios):**
    *   **Express.js (Convención):**
        *   **Controladores (Handlers de Ruta):** Las funciones que se pasan como segundo argumento a `app.get()`, `app.post()`, etc. (ej: `(req, res) => { /* lógica aquí */ }`).
        *   **Servicios (Opcional, pero buena práctica):** Para lógica más compleja, es común separar esta lógica en módulos/funciones aparte (a menudo llamados "servicios") que los controladores invocan. Esto mejora la organización y la testeabilidad.
            ```javascript
            // controllers/userController.js
            const userService = require('../services/userService');
            exports.createUser = async (req, res) => {
                try {
                    const newUser = await userService.create(req.body);
                    res.status(201).json(newUser);
                } catch (error) {
                    res.status(400).json({ message: error.message });
                }
            };
            ```

5.  **Acceso y Manipulación de Datos (Modelos/Capa de Datos):**
    *   **Express.js:** No impone una forma específica de interactuar con bases de datos. Esto depende de la base de datos que elijas y de las librerías que uses.
        *   **ORMs/ODMs:**
            *   **Sequelize (SQL):** Para bases de datos como PostgreSQL, MySQL, SQLite, MSSQL. Define modelos y proporciona métodos para interactuar con la BD.
            *   **Mongoose (MongoDB):** Para MongoDB. Define esquemas y modelos, ofreciendo una capa de abstracción sobre el driver nativo de MongoDB.
            *   **Prisma, TypeORM:** Otras alternativas populares.
        *   **Drivers Nativos:** Puedes usar los drivers directos (ej: `pg` para PostgreSQL, `mongodb` para MongoDB) si prefieres más control, aunque es más verboso.
        *   Los modelos (definidos con el ORM/ODM) encapsulan la lógica de interacción con la base de datos.

6.  **Autenticación y Autorización:**
    *   **Express.js:** Se implementa comúnmente usando middleware.
        *   **Estrategias:** JWT (JSON Web Tokens) es muy popular para APIs. También se usan sesiones (con librerías como `express-session`).
        *   **Librerías:** `passport.js` es una librería de autenticación extremadamente flexible y popular que se integra muy bien con Express mediante estrategias (ej: `passport-jwt`, `passport-local`, `passport-oauth2`).
        *   El middleware de autenticación verifica el token/sesión y, si es válido, a menudo añade la información del usuario al objeto `req` (ej: `req.user`).
        *   La autorización se implementa luego, a menudo en otro middleware o dentro del controlador, verificando roles o permisos del `req.user`.

7.  **Manejo de Errores:**
    *   **Express.js:**
        *   Si un error ocurre en código síncrono dentro de un handler de ruta o middleware, Express lo captura por defecto si no hay un `try...catch`.
        *   Para errores en código asíncrono (promesas, async/await), debes asegurarte de capturarlos (con `.catch()` o `try...catch` en `async` functions) y pasar el error a `next(error)`.
        *   **Middleware de manejo de errores centralizado:**
            ```javascript
            app.use((err, req, res, next) => {
                console.error(err.stack);
                res.status(err.status || 500).send(err.message || 'Something broke!');
            });
            ```
            Este debe ser el último `app.use()` que definas.

8.  **Validación de Datos:**
    *   **Express.js:** Se puede hacer manualmente, pero es mejor usar librerías.
        *   **Librerías Populares:** `joi`, `express-validator`, `yup`.
        *   A menudo se implementa como un middleware antes del controlador de la ruta. Si la validación falla, el middleware de validación envía una respuesta de error (ej: 400 Bad Request) y no llama a `next()`.

9.  **Configuración:**
    *   **Node.js/Express.js:**
        *   **Variables de Entorno:** Es la práctica recomendada. Usar el módulo `dotenv` para cargar variables desde un archivo `.env` en desarrollo. En producción, estas variables se configuran directamente en el entorno del servidor.
        *   Archivos de configuración (`config.js`): Pueden leer variables de entorno y exportar un objeto de configuración.
        *   `app.get('env')` devuelve el entorno actual (por defecto 'development' si `NODE_ENV` no está seteado).

10. **Seguridad (Específico de Express):**
    *   **Librerías:**
        *   `helmet`: Ayuda a proteger tu aplicación de algunas vulnerabilidades web conocidas configurando apropiadamente cabeceras HTTP.
        *   `csurf` (o `csurf-lusca`): Protección contra CSRF (Cross-Site Request Forgery), más relevante si usas sesiones y formularios tradicionales.
        *   `express-rate-limit`: Para limitar la tasa de peticiones y prevenir ataques de fuerza bruta.
    *   **Prácticas:** Validar todas las entradas, usar consultas parametrizadas o ORMs para prevenir inyección SQL, escapar salidas para prevenir XSS (aunque si sirves JSON a un frontend SPA, el frontend es el principal responsable de escapar).

11. **Logging y Monitoreo:**
    *   **Node.js/Express.js:**
        *   **Logging:**
            *   `console.log()` (simple, para desarrollo).
            *   `morgan`: Middleware popular para logging de peticiones HTTP en formato Apache.
            *   Librerías más robustas como `winston` o `pino` para logging estructurado, con niveles y diferentes transportes (consola, archivo, servicios externos).
        *   **Monitoreo:** Herramientas como PM2 (para gestionar procesos Node.js, incluye monitoreo básico), o integración con servicios de APM (Application Performance Monitoring) como New Relic, Datadog, Sentry (más enfocado en errores).

Esta lista cubre los componentes esenciales. La estructura de carpetas de un proyecto Express también es importante para la organización, aunque Express es muy flexible y no impone una estructura rígida (comúnmente se ven carpetas como `routes`, `controllers`, `services`, `models`, `middlewares`, `config`, `public`).
